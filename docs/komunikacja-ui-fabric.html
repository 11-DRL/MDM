4Login flow (OAuth 2.0 Authorization Code + PKCE)

1 Browser (MSAL) generuje code_verifier + code_challenge (SHA256)
2 GET /authorize z code_challenge, client_id, redirect_uri, scope
3 Entra ID wyświetla login (user + MFA)
4 Consent raz na scope
5 Redirect 302 do SWA z ?code=ABC&state=XYZ
6 POST /token z code + code_verifier
7 Entra waliduje PKCE → zwraca {id_token, access_token, refresh_token}
8 MSAL zapisuje tokeny w sessionStorage
9 UI renderuje GoldenList
🛡️ Klucz bezpieczeństwa: secrets nigdy nie są w kodzie SPA. PKCE gwarantuje że ktoś kto przechwyci code bez code_verifier nie wymieni go na token.

6Walidacja tokenu w Function — kod

// azure-function/src/functions/mdmWrite.ts
export async function mdmWrite(req, context) {
  // 1. EasyAuth zwalidował token i wstrzyknął nagłówki
  const userOid = req.headers["x-ms-client-principal-id"];
  const userUpn = req.headers["x-ms-client-principal-name"];
  const principal = JSON.parse(
    Buffer.from(req.headers["x-ms-client-principal"], "base64").toString()
  );
  const roles = principal.claims
    .filter(c => c.typ === "roles").map(c => c.val);

  // 2. RBAC
  if (!roles.includes("Steward"))
    return { status: 403, body: "Brak roli Steward" };

  // 3. Walidacja payloadu
  if (!["approve_match","reject_match","edit_golden","new_location"]
        .includes(req.body.action))
    return { status: 400, body: "Nieznana akcja" };

  // 4. Połączenie z Fabric przez Managed Identity
  const cred = new DefaultAzureCredential();
  const token = await cred.getToken("https://database.windows.net/.default");
  const conn = await sql.connect({
    server: process.env.FABRIC_SQL_ENDPOINT,
    database: process.env.FABRIC_WAREHOUSE_NAME,
    authentication: {
      type: "azure-active-directory-access-token",
      options: { token: token.token }
    }
  });

  // 5. Transakcja + audit
  const tx = conn.transaction();
  await tx.begin();
  try {
    await executeAction(tx, req.body, userOid);
    await tx.request()
      .input("user_oid", userOid)
      .input("user_upn", userUpn)
      .input("action", req.body.action)
      .query(`INSERT INTO fact_steward_audit 
              (user_oid, user_upn, action, timestamp)
              VALUES (@user_oid, @user_upn, @action, SYSUTCDATETIME())`);
    await tx.commit();
    return { status: 200, body: { ok: true } };
  } catch (e) {
    await tx.rollback();
    throw e;
  }
}

7Token lifecycle

Typ tokenuGdzieLifetimeRefresh
id_token (user) browser 1h refresh_token → /token
access_token (→ Function) browser 1h MSAL silent renew (iframe, prompt=none)
MI access_token (→ Fabric) Function env 24h auto (DefaultAzureCredential)
refresh_token browser SS 24h / 90d sliding, nowy przy użyciu
Silent renewal w SPA: MSAL.js robi acquireTokenSilent() 5 min przed expiry. Jeśli refresh_token ważny → bez interakcji. Jeśli expired → popup/redirect.

📌 TL;DR

  • UI → Entra: OAuth2 Authorization Code + PKCE (MSAL.js)
  • UI → Function: JWT Bearer token (audience = Function)
  • Function → Fabric: Managed Identity (audience = database.windows.net)
  • Audit: Function wyciąga user_oid z tokenu UI, zapisuje w Fabric
  • Zero sekretów w UI (SPA) ani w Function (MI zamiast connection string)
  • RBAC: user ma app role "Steward" w Entra; MI Function ma Contributor w Fabric workspace
